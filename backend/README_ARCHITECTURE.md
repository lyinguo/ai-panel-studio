# 架构核心原则

> **多个独立讨论并发，各自状态、事件流、数据严格互不干扰。**

## 原则阐述

本系统最底层的设计约束是：**隔离（Isolation）**。系统必须原生支持同时运行多个完全独立的讨论会话（Session），每个会话拥有自己封闭的上下文环境，彼此之间在以下三个维度严格隔离：

### 1. 状态隔离

- 每个讨论拥有独立的状态快照，包含但不限于：对话历史、当前 Agent 执行上下文、临时变量、UI 交互状态。
- 任何会话的状态变更不得泄漏、级联或影响到其他会话。
- 后端不应在内存中维持跨会话的共享可变状态；如需共享（如用户信息、全局配置），必须通过显式的只读查询访问持久化存储。

### 2. 事件流隔离

- 每个讨论拥有独立的 Server-Sent Events（SSE）流，事件推送范围为该会话内部。
- 一个会话产生的事件（Token 流、Agent 状态变更、错误通知等）仅向该会话的订阅者推送。
- 事件路由必须携带 Session ID，服务层按 Session ID 进行过滤和分发。

### 3. 数据隔离

- 所有持久化数据必须携带会话归属标识（`session_id` 或等价外键）。
- 数据查询必须按会话范围过滤，禁止跨会话读取。
- 数据库表设计需满足：给定 `session_id`，查询结果集绝不包含其他会话的行。

## 设计推论

上述原则推导出以下架构决策：

| 领域 | 决策 |
|---|---|
| **路由** | 每个 API 请求必须携带 `session_id` 路径参数或 Header |
| **数据库** | 核心表以 `session_id` 作为分片键 / 聚簇索引前缀 |
| **事件** | SSE 端点以 `session_id` 区分 Event Stream |
| **缓存** | 缓存键必须以 `session_id` 作为前缀 |
| **中间件** | 请求级中间件注入当前 Session 上下文，业务层不感知多会话存在 |

## 违反示例

```
❌ 全局变量存储当前讨论的 Token 计数（并发时相互覆盖）
❌ SSE 推送未按 session_id 过滤（用户 A 看到用户 B 的流）
❌ 查询会话消息未加 WHERE session_id = ?（泄漏其他会话数据）
❌ 内存中共享的 Agent 实例池（状态互相污染）
```

## 检查清单

实现任何新功能前，请确认：

- [ ] 是否引入了跨会话的共享内存状态？
- [ ] 新 API 是否要求/验证了 `session_id`？
- [ ] 新数据库查询是否按 `session_id` 过滤？
- [ ] 新事件推送是否仅发送给目标会话的订阅者？
- [ ] 如果该功能只运行一个会话时正常，运行两个会话时是否仍正常？

---

**违反这一原则等于违反系统设计。如果无法在约束内实现功能，应重新审视方案而非绕过隔离。**
